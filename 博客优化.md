# 博客优化记录

## 1. 文档目的

这份文档用于记录当前 `MkDocs + Material for MkDocs` 个人博客项目的结构诊断结果、可精简项、目录职责说明，以及后续长期维护时推荐遵循的管理规范。

这次优化的目标不是立刻“大拆大改”，而是先把项目分成 4 类内容：

1. 真正需要保留的博客源码。
2. 主题定制必须依赖的覆盖文件。
3. 可以迁移出当前仓库主线的历史资料、实验文件、缓存和构建产物。
4. 已确认存在引用异常、重复、命名混乱或职责不清的问题点。

---

## 2. 当前项目现状概览

当前仓库整体上已经具备可用的博客形态，但有一个很明显的问题：

> “博客源码仓库”同时承载了博客内容、主题覆盖、实验脚本、简历资料、缓存、IDE 配置、临时测试文件、可执行文件、字体压缩包等多类内容。

这会导致 5 个直接后果：

1. 根目录职责不清，后续加东西时只能“往里面塞”。
2. `docs` 目录既放内容页，也放静态资源、主题覆盖、个人资料和非博客附件，边界模糊。
3. `mkdocs.yml` 配置过长，保留了大量 fork 模板遗留配置、注释和外链资源。
4. 仓库里混入缓存、IDE 元数据、Python 字节码、历史测试文件，影响可维护性。
5. 某些页面、脚本、样式和导航已经出现“引用路径不一致”或“配置里写了但实际文件不存在”的问题。

---

## 3. 这次梳理中确认发现的问题

### 3.1 已确认的结构性问题

#### 问题 1：仓库中混入明显不应长期提交的文件

当前仓库内存在以下典型“非源码”内容：

- `.cache/`
- `.ai_cache/`
- `docs/overrides/hooks/__pycache__/`
- `code/leetcode/.idea/`
- `code/hello.exe`

这类文件的问题：

- 不是博客内容源码。
- 不是主题必要资源。
- 会污染版本历史。
- 会让仓库看起来像“开发现场快照”，而不是“可维护的博客项目”。

#### 问题 2：根目录存在重复/过时工作流配置

当前同时存在：

- `.github/workflows/PublishMySite.yml`
- `PublishMySite.yml`

其中根目录这个 `PublishMySite.yml` 文件内容已经明确写了“此工作流已经被弃用”，因此它属于典型的历史残留文件。

#### 问题 3：`mkdocs.yml` 过长且职责混杂

当前 `mkdocs.yml` 同时承担了：

- 站点基础配置
- 导航配置
- 主题功能配置
- 大量中文注释
- 插件配置
- 第三方资源引用
- 自定义 hooks 注册
- 主题外链样式覆盖

问题不在“长”，而在“不可控”：

- 配置里夹杂 fork 主题原作者遗留内容。
- 保留了大量暂时未启用但未清理的插件注释。
- 有重复键和尾部残留痕迹。
- 很难一眼看出哪些配置是你真正需要的。

#### 问题 4：内容目录与资源目录边界混乱

当前 `docs/` 下混合了：

- 博客正文
- 个人资料页
- PDF 附件
- 字体文件
- 样式文件
- JS 文件
- 模板覆盖
- hook 脚本

这会使后续维护时很难回答以下问题：

- 某个文件是“页面内容”还是“构建资源”？
- 某个 PDF 是页面附件还是简历下载文件？
- 某个 CSS 是主题全局样式还是某一页的局部样式？

#### 问题 5：`code/` 目录和博客主线耦合不清

当前根目录下的 `code/` 包含：

- `gzh/`
- `leetcode/`
- `optimize/`
- `hello.cpp`
- `hello.exe`

其中有些是实验代码，有些是算法代码，有些是单独小项目，有些只是临时文件。它们可以存在，但不应继续以“无明确规则的堆放方式”增长。

---

### 3.2 已确认的引用与配置问题

#### 问题 6：`mkdocs.yml` 中存在引用了但实际缺失的资源

已经确认缺失的引用包括：

- `about/test.md`
- `javascripts/简历.js`
- `stylesheets/简历.css`

说明：

- `about/test.md` 被写进导航，但仓库内不存在。
- `javascripts/简历.js` 在 `extra_javascript` 中被引用，但不存在。
- `stylesheets/简历.css` 在 `extra_css` 中被引用，但不存在。

这类问题会直接影响构建稳定性和配置可信度。

#### 问题 7：MathJax 脚本路径组织不统一

当前实际存在的 JS 文件分布不一致：

- 根目录下有 `javascripts/config_mathjax.js`
- `docs/javascripts/` 下有 `mathjax-config.js`、`mathjax.js`、`extra.js` 等

但 `mkdocs.yml` 中写的是：

- `javascripts/extra.js`
- `javascripts/config_mathjax.js`

这会造成一个明显问题：

- 一部分 JS 被放在 `docs/javascripts/`
- 一部分 JS 被放在根目录 `javascripts/`

从维护角度看，这是不规范的。对于 `MkDocs` 项目，静态资源更适合统一收敛到 `docs/assets/`、`docs/javascripts/`、`docs/stylesheets/` 这类固定位置。

#### 问题 8：首页和个人简介页的样式引用路径可疑

在以下页面中发现了相同写法：

- `docs/index.md`
- `docs/about/geren.md`

都包含：

```html
<link rel="stylesheet" href="../sty/portfolio.css">
```

但实际样式文件位于：

- `docs/about/sty/portfolio.css`

这说明页面引用路径与文件放置位置不匹配，至少从结构设计上是混乱的，极大概率属于历史挪动后没有统一修正的遗留问题。

#### 问题 9：`mkdocs.yml` 尾部存在明显残留痕迹

在 hooks 配置尾部，出现了注释和配置粘连的残留内容，后面又重复写了站点字段。这虽然不一定立刻导致解析失败，但说明配置文件已经经历过多次手工拼接，建议尽快清理。

#### 问题 10：部分 hook 脚本和测试脚本未形成明确边界

当前 `docs/overrides/hooks/` 下除了正式 hook 外，还存在：

- `test.py`
- `test2.py`

如果它们没有被 `mkdocs.yml` 注册，也不是文档中明确说明的开发测试工具，那么它们不应该长期留在正式主题覆盖目录中。

## 4. 当前目录的作用解释

下面按“这个项目里最关键的目录”来解释其职责、原理和最佳实践。

### 4.1 根目录

根目录应该只保留“站点级入口文件”和“仓库级配置”。

当前根目录中真正合理的核心文件通常应包括：

- `mkdocs.yml`
- `README.md`
- `requirements.txt`
- `.gitignore`
- `.github/`
- `docs/`
- `LICENSE`

根目录不适合长期堆放：

- 临时脚本
- 缓存目录
- 过时工作流
- 可执行文件
- 与博客站点无关的小实验二进制

最佳实践：

- 根目录要尽量“像一个静态站点项目”，而不是“通用资料收纳箱”。
- 非站点主线内容应迁移到单独仓库，或至少移动到边界清晰的子目录中。

---

### 4.2 `docs/`

`docs/` 是 MkDocs 的默认文档根目录。

原理上，MkDocs 会把这个目录当作“站点源文件树”：

- Markdown 页面从这里读取。
- 静态文件也从这里复制到最终站点。
- 图片、CSS、JS、PDF 等资源如果放在这里，构建时会一起参与输出。

所以 `docs/` 的本质不是“随便放点博客文件的目录”，而是：

> 最终站点可见资源的源码根目录。

最佳实践：

- 只放“站点页面”和“站点运行时需要的静态资源”。
- 不要把开发缓存、临时测试、编辑器产物混进来。
- 尽量把页面内容和静态资源分层组织。

推荐再细分成：

- `docs/index.md`
- `docs/about/`
- `docs/blog/`
- `docs/assets/`
- `docs/stylesheets/`
- `docs/javascripts/`
- `docs/overrides/`

---

### 4.3 `docs/blog/`

这是博客正文内容目录，应该是你的核心内容资产。

它现在承担的作用包括：

- 技术文章
- 算法题记录
- 机器学习笔记
- 随笔
- 每日诗词
- 特殊页面

这个目录本身没有问题，问题在于内部命名层级和内容类型不够统一。

当前观察到的情况：

- 英文目录名、中文目录名混用。
- “专题栏目”和“资料附件”混在同层。
- `fonts/` 这种资源目录出现在 `blog/` 下，不够自然。
- `Algorithm/算法基础/code/` 下收纳了大量源码和图片素材，像是一个独立知识库项目。

最佳实践：

- `blog/` 只承载“博客内容领域结构”。
- 页面附件尽量靠近文章，但公共资源不要放在 `blog/` 根层。
- 公共字体、公共脚本、全局样式不要放在 `blog/` 下。

建议后续改成更稳定的栏目结构，例如：

- `docs/blog/computer-science/`
- `docs/blog/machine-learning/`
- `docs/blog/notes/`
- `docs/blog/poems/`

如果你不想大改英文命名，至少也要统一成“一个栏目一种风格”，不要中英来回切。

---

### 4.4 `docs/about/`

这是个人介绍类页面目录，定位应该是：

- 个人简介
- 简历页面
- 支持作者
- 长期主义等个人页面

这个目录的定位是合理的，但当前有 3 个问题：

1. PDF 文件重复较多。
2. 里面混入了 `sty/portfolio.css` 这样的局部样式资源。
3. 页面和附件没有建立清晰的命名规范。

例如当前已经出现多份简历相关 PDF：

- `docs/about/个人简历.pdf`
- `docs/about/龚子航_SLAM算法工程师_简历.pdf`
- `docs/about/resume/龚子航_SLAM算法工程师_简历.pdf`
- `docs/overrides/龚子航_SLAM算法工程师_简历.pdf`

这会造成后续你自己都分不清：

- 哪份是最新的？
- 哪份被页面实际引用了？
- 哪份只是备份？

最佳实践：

- `about/` 下面页面和附件分开。
- 附件单独放 `about/files/` 或 `about/assets/`。
- 简历只保留一个正式版本，其他版本不要散落多处。

---

### 4.5 `docs/assets/`

这个目录是最适合放“公共静态资源”的地方。

原理上它适合承载：

- 图片
- 字体
- 第三方库静态文件
- 页面公共附件
- 某些插件依赖的本地资源

当前仓库中 `docs/assets/document_dates/` 是一个典型的插件资源目录，这种组织方式是合理的。

最佳实践：

- 公共资源统一放这里。
- 插件专用资源单独子目录归档。
- 页面私有资源才考虑跟页面放在一起。

你现在 `blog/fonts/`、`about/sty/` 这些目录，其实都更适合迁移到 `assets/` 或 `stylesheets/`。

---

### 4.6 `docs/stylesheets/`

这是全局样式覆盖目录。

当前已有：

- `extra.css`
- `extra1.css`
- `extra2.css`
- `customize.css`
- `link.css`
- `portfolio.css`
- `shubiao.css`
- `video.css`
- `ziti.css`

问题不是文件多，而是命名没有表达职责：

- `extra.css`、`extra1.css`、`extra2.css` 从文件名上几乎无法判断作用。
- `customize.css` 语义过宽。
- `portfolio.css` 既出现在这里，也在 `about/sty/` 有同类资源概念。

最佳实践：

- 文件名直接表达用途。
- 全局样式按功能拆分，而不是按时间顺序命名。

推荐命名方式：

- `base.css`
- `theme-overrides.css`
- `resume.css`
- `components.css`
- `search.css`
- `font.css`

---

### 4.7 `docs/javascripts/`

这是前端脚本目录。

当前已有：

- `extra.js`
- `mathjax-config.js`
- `mathjax.js`
- `backgroud.js`
- `rellax.min.js`
- `shortcuts.js`
- `shubiao.js`
- `view.js`

问题在于：

- 根目录又单独放了 `javascripts/config_mathjax.js`
- 命名风格也不统一
- 看不出哪些是正式启用的，哪些只是历史尝试

最佳实践：

- 所有站点脚本统一只放一处。
- 第三方压缩库单独归类。
- 自写脚本按功能命名。

推荐结构：

- `docs/javascripts/app/`
- `docs/javascripts/vendor/`
- `docs/javascripts/math/`

---

### 4.8 `docs/overrides/`

这是最关键的主题定制目录之一。

原理上，`theme.custom_dir: docs/overrides` 表示：

- 这里的模板文件会覆盖 Material 主题的默认模板。
- `partials/` 通常用于局部模板复写。
- 自定义页面模板也可以放这里。
- hook 脚本虽然能放这里，但本质上属于“构建逻辑代码”，不是模板资源。

当前目录中包含：

- `partials/`
- `404.html`
- `health.html`
- `resume-template.html`
- `朋友圈.html`
- `随笔.html`
- `hooks/`
- 一个 PDF 文件

其中需要明确的是：

- HTML 模板属于 overrides 的正常用途。
- `hooks/` 放在这里可以运行，但从职责划分上不够优雅。
- PDF 文件放在这里不合适，因为它不是模板，也不是 partial。

最佳实践：

- `overrides/` 只放模板覆盖相关文件。
- 非模板静态文件不要往这里塞。
- hooks 最好迁移到单独目录，如 `tools/hooks/` 或 `scripts/mkdocs_hooks/`。

---

### 4.9 `docs/overrides/hooks/`

这里目前承担的是 MkDocs 构建 hook 逻辑，包括：

- `socialmedia.py`
- `reading_time.py`
- `ai_summary.py`

这类脚本的本质是“构建时 Python 插件逻辑”，不是主题模板。

所以它们虽然能工作，但从工程结构上看，更合理的位置是：

- `scripts/mkdocs_hooks/`
- `tools/mkdocs_hooks/`
- `extensions/`

当前存在的问题：

- 目录中混入 `__pycache__/`
- 有 `test.py`、`test2.py` 这样的临时文件
- AI 摘要 hook 逻辑很长，职责偏重，配置与缓存策略复杂

最佳实践：

- hooks 单独设目录。
- 每个 hook 一个清晰职责。
- 测试文件不要混在正式 hook 目录。
- 缓存策略、环境变量、是否在本地启用，都应该有单独说明文档。

---

### 4.10 `.github/`

这个目录负责 GitHub 平台层面的自动化与协作配置。

当前包含：

- `workflows/PublishMySite.yml`
- `ISSUE_TEMPLATE/`
- `FUNDING.yml`

如果你当前只是自己维护个人博客，而不打算公开接收 issue/funding，那么：

- `ISSUE_TEMPLATE/`
- `FUNDING.yml`

这两类文件不是必须保留。

最佳实践：

- 个人博客仓库只保留你实际在用的 GitHub 配置。
- 不要为了“看起来像模板项目”而保留无用平台配置。

---

### 4.11 `code/`

这是当前仓库中最容易继续失控的目录。

如果你希望博客仓库只做“博客与展示”，那么 `code/` 应该扮演以下两种角色中的一种：

#### 角色 A：文章配套源码仓库内镜像

这种情况下：

- 代码是为了文章配套展示。
- 每个子目录必须能明确对应文章主题。
- 不要出现无关临时二进制和 IDE 配置。

#### 角色 B：独立项目集合

这种情况下：

- 它已经不是博客附属目录，而是“多项目收纳区”。
- 更适合拆成单独仓库，或至少单独放到 `projects/` 目录，并写清楚每个项目用途。

当前 `code/` 的问题是介于两者之间，边界不清。

---

## 5. 哪些内容可以删掉

下面按优先级给出建议。

### 5.1 可以直接删除的内容

这类内容通常不会影响博客正文和主题运行。

#### A. 明显缓存或构建中间产物

- `.cache/`
- `docs/overrides/hooks/__pycache__/`

#### B. 已弃用或重复的配置文件

- 根目录 `PublishMySite.yml`

#### C. IDE / 本地产物

- `code/leetcode/.idea/`

#### D. 临时或无意义的二进制文件

- `code/hello.exe`

#### E. 如果确认未使用的 hook 测试文件

- `docs/overrides/hooks/test.py`
- `docs/overrides/hooks/test2.py`

---

### 5.2 建议删除或迁移的内容

这类内容需要你先确认“是否还在实际使用”，再决定删或搬。

#### A. `.ai_cache/`

是否删除，取决于你的 AI 摘要策略：

- 如果你确实依赖缓存加速 CI，可以保留。
- 如果你已经不打算继续维护 AI 摘要自动生成，建议停止提交并清理。

注意：

当前 `.gitignore` 明确写了要保留 `.ai_cache/`，这说明它曾经被当作正式项目资源管理，而不是临时缓存。是否保留，需要你主动做一次产品决策。

#### B. 重复 PDF

建议只保留“最终被页面链接使用的唯一正式版本”，其余全部迁出或删除。

重点检查：

- `docs/about/`
- `docs/about/resume/`
- `docs/overrides/`
- `docs/blog/`

#### C. `docs/blog/fonts/`

这里包含：

- `.ttf`
- `.zip`
- 示例图片
- 授权说明 PDF

如果这些字体只是为了某一篇特殊页面使用，可以迁移到更明确的位置，例如：

- `docs/assets/fonts/`

如果只是历史尝试且不再使用，可以删除非必要备份、压缩包和重复字体文件。

#### D. `code/hello.cpp`

如果没有任何文章引用，也不是项目的一部分，建议删除。

#### E. `code/leetcode/`

如果只是个人刷题草稿，建议不要继续放在博客主仓库；
如果你想保留，建议迁移到单独仓库或至少改造成“按题目/语言组织”的规范目录。

---

### 5.3 目前不建议直接删除的内容

这些内容虽然看起来“杂”，但仍可能是博客正在使用的核心功能。

- `docs/overrides/partials/`
- `docs/overrides/resume-template.html`
- `docs/overrides/health.html`
- `docs/overrides/hooks/socialmedia.py`
- `docs/overrides/hooks/reading_time.py`
- `docs/overrides/hooks/ai_summary.py`
- `docs/assets/document_dates/`

## 7. `mkdocs.yml` 的最佳管理规范

### 7.1 配置文件应按块组织

建议固定按下面顺序组织：

1. 基础站点信息
2. 主题配置
3. 导航配置
4. 插件配置
5. Markdown 扩展
6. 静态资源
7. hooks
8. extra

### 7.2 不要把“实验配置”长期注释堆在正式配置里

当前 `mkdocs.yml` 里保留了很多暂时不用的注释配置。短期看方便，长期看会让配置难以辨认。

建议：

- 真正停用且短期不会恢复的配置，直接删除。
- 想保留参考，就放进 `docs/notes/mkdocs-config-notes.md` 或单独备份文档。

### 7.3 静态资源引用必须统一目录规则

建议明确规则：

- CSS 统一来自 `docs/stylesheets/`
- JS 统一来自 `docs/javascripts/`
- 图片/字体/PDF 统一来自 `docs/assets/`

不要再出现：

- 一部分脚本在根目录
- 一部分脚本在 `docs/`
- 一部分字体在 `blog/fonts/`
- 一部分样式在 `about/sty/`

### 7.4 导航不要纯手工“越写越长”

目前 `nav` 已经很长，后续维护成本会继续增加。

建议：

- 栏目层级保持 2 到 3 层即可。
- 大专题才手工维护导航。
- 高频新增文章的栏目可考虑依赖 blog 插件能力，而不是所有文章都强行手写到 `nav` 中。

### 7.5 hooks 和插件的职责要尽量单一

建议定义成：

- `reading_time.py` 只负责阅读信息
- `socialmedia.py` 只负责分享按钮
- `ai_summary.py` 只负责摘要

并为每个 hook 单独写说明，至少记录：

- 功能
- 依赖环境变量
- 启用条件
- 本地/CI 行为差异

## 8. 推荐的日常维护规范

### 8.1 新增文章时

遵循这个顺序：

1. 先确定文章属于哪个栏目。
2. 页面文件只放到对应栏目。
3. 页面私有图片放文章同级 `images/` 或统一 `assets/images/`。
4. 不要顺手往根目录或 `docs/` 根层扔附件。

### 8.2 新增样式时

先判断属于哪类：

- 全站通用样式
- 某一专题通用样式
- 某单页私有样式

然后再决定放：

- `docs/stylesheets/`
- `docs/assets/`
- 页面内嵌少量样式

不要再新增 `extra3.css`、`final.css`、`new.css` 这类时间型命名。

### 8.3 新增脚本时

统一放入 `docs/javascripts/`，并写清楚是：

- 页面交互脚本
- 数学公式脚本
- 第三方库
- 动效脚本

### 8.4 新增下载附件时

统一归档：

- 简历类放 `docs/about/files/`
- 博文附件放 `docs/assets/pdf/` 或文章同级附件目录

不要在 `about/`、`blog/`、`overrides/` 多处同时放相似 PDF。

### 8.5 新增 hook 或自动化逻辑时

先判断：

- 这是主题模板覆盖问题？
- 这是构建阶段逻辑问题？

如果是构建逻辑，就不要继续塞进 `overrides/`。
